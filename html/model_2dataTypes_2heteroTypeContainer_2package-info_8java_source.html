2 This package revolves around the HeteroFieldContainer and its associated classes.

4 # Introduction {#HETERO_TYPE_CONTAINER_introduction}
5 This is a typesafe container that can hold heterogenous types as defined by a user defineable schema.
6 This container is useful when dealing with return values from database queries, as you can use the
7 framework surrounding this datatype to define a database schema and access the return values, in their
8 native form, using a field type rather than an indexed list or Map of homogenous types.
9 
10 This type is also more widely applicable but I suggest to use this type when it is the ONLY viable
11 option within your particular problem domain. It also requires a small ecosystem of classes, so again
12 this is probably not the right choice as a general purpose datatype.
13 
14 Its based around a modification of the typesafe hetereogeneous container pattern described in Joshua
15 Bloch's book 'Effective Java' (second edition, Item 29). For a description of the pattern and an
16 example see http://www.codeaffine.com/2015/03/04/map-distinct-value-types-using-java-generics/
17 
18 # Usage {#HETERO_TYPE_CONTAINER_usage}
19 In order to use the HeteroTypeContainer, there is a little ecosystem of classes that must be set up
20 beforehand so that the heterogenous type container will work properly
21 
22 ## 1.define field types{#HETERO_TYPE_CONTAINER_types}
23 The first thing that must be done is to define the field types that you are going to wish stored in
24 the container. The purpose of this is to define meaningful fields that will ultimately fit together
25 in a consistant data schema.
26 
27 A field is defined by implementing the AbstractField interface. The example below shows an example
28 of a 'Name' field which could relate to an address table in a database or student record object:
29 
30 ~~~{.java}
31 public final class Name implements AbstractField
32 {
33  //this int should define the maximum or nominal size of this field depending on what makes sense given the users data
34  private static final int size_M = 50;
35 
36  //default ctor
37  public Name()
38  {super();}
39 
40  //this method defines the fundemental 'Type' of this field. This, ultimately, must correspond
41  //to the actual type of the data, so as this field represents a name it is textual data and
42  //will be represented by a String. If this field represented an age then 'TypeDef.BYTE' or
43  //'TypeDef.SHORT' would be more appropriate etc.
44  @Override
45  public String type()
46  {return TypeDef.STRING;}
47 
48  //this method defines a size for this field, this is not directly used by the HeteroFieldContainer
49  //but can be useful for the user as most data should be of a certain size, using this method the
50  //user can retrieve this value for checking purposes.
51  @Override
52  public int size()
53  {return Id.size_M;}
54 }
55 ~~~
56 
57 In the `type()` method you will notice the use of the `TypeDef` class, this class should be used when
58 defining the type of a particular field. `TypeDef` contains a bunch of `public static final String variables`
59 that contain the canonical name of a specific type, the HeteroFieldContainer uses this during its
60 type checking in order to make sure that the data being added/retrieved is of the same type as the
61 field that is supposed to represent it.
62 
63 This is quite a simple procedure but necessary to the functioning of the HeteroFieldContainer. Once
64 you have defined all the fields that make up your data schema you can structure the schema.
65 
66 ## 2.define field schema{#HETERO_TYPE_CONTAINER_schema}
67 Strictly speaking, this part is optional although thoroughly recommended. It is possible to use the
68 HeteroFieldContainer with the raw fields, passing a `new Name()` (in our example), however to save
69 creating a great many instances that do not really need creating it is best to define a schema.
70 
71 Included in this library is an empty interface (AbstractTypeSchema) it is not necessary to use this
72 interface specifically but it can help clarify and document your intentions. Below is an example of
73 a data schema.
74 
75 ~~~{.java}
76 public final class BuyerTable implements AbstractTypeSchema
77 {
78  //The buyerID column of the buyers table
79  // - VARCHAR(40)
80  // - Primary Key
81  public static final BuyerID BUYERID = new BuyerID();
82  //The name column of the buyers table
83  // - VARCHAR(45)
84  public static final Name NAME = new Name();
85  //The street1 column of the buyers table
86  // - VARCHAR(80)
87  public static final Street STREET1 = new Street();
88  //The street2 column of the buyers table
89  // - VARCHAR(80)
90  public static final Street STREET2 = new Street();
91  //The city column of the buyers table
92  // - VARCHAR(80)
93  public static final City CITY = new City();
94  //The county column of the buyers table
95  // - VARCHAR(80)
96  public static final County COUNTY = new County();
97  //The postcode column of the buyers table
98  // - VARCHAR(15)
99  public static final Postcode POSTCODE = new Postcode();
100  //The email column of the buyers table
101  // - VARCHAR(100)
102  public static final Email EMAIL = new Email();
103  //The phoneNo column of the buyers table
104  // - BIGINT(16)
105  public static final Phone PHONE = new Phone();
106 
107  //private ctor ensures this class is NEVER INSTANTIATED
108  private BuyerTable()
109  {super();}
110 }
111 ~~~
112 
113 The above class represents the buyer table of a database and as such is part of a data schema rather
114 than all of it. While not strictly necessary for the use of the container it aids in both efficient
115 usage of the container as well as allowing you to formalise and document your data schema within the
116 program code, making it easier to modify and maintain, as well as making your intentions clear when
117 using the container.
118 
119 ## 3.use container{#HETERO_TYPE_CONTAINER_use}
120 The usage of the container itself is relatively straightforward; after creating a new instance of the
121 container:
122 
123 ~~~{.java}
124 HeteroFieldContainer containerInstance = new HeteroFieldContainer();
125 ~~~
126 
127 it can be populated with data:
128 
129 ~~~{.java}
130 containerInstance.add(BuyerTable.NAME, "Jeff the plumber");
131 containerInstance.add(BuyerTable.BUYERID, "jeff43");
132 containerInstance.add(BuyerTable.STREET1, "12 some apartement building");
133 containerInstance.add(BuyerTable.STREET2, "some street");
134 containerInstance.add(BuyerTable.COUNTY, "a county");
135 containerInstance.add(BuyerTable.CITY, "cityville");
136 containerInstance.add(BuyerTable.POSTCODE, "AB12 9ZX");
137 containerInstance.add(BuyerTable.EMAIL, "jeff@plumber.co.uk");
138 containerInstance.add(BuyerTable.PHONE, "441234567890");
139 ~~~
140 
141 The container does not need to have all the fields in a schema, and they do not have to be in any
142 particular order, however there is a restriction that the container can only contain one instance of
143 any particular field.
144 
145 In order to retrieve data from the container, the following procedure is followed:
146 
147 ~~~{.java}
148 String name = containerInstance.get(BuyerTable.NAME, ClassRef.STRING);
149 long phone = containerInstance.get(BuyerTable.PHONE, ClassRef.INTEGER);
150 ~~~
151 
152 in the above example the Name field and the Phone field are retrieved from the HeteroFieldContainer.
153 You will notice the `ClassRef` class being used as a type identifier for the container, this is
154 similar to the `TypeDef` class in the sense that it provides a series of `public static final` variables
155 of type `Class<T>`. The reasoning behind this is that in a container which can contain any number of
156 arbitrary types in order to aid in safety the container enforces that the user should know the type of
157 the field that they are asking for.
158 
159 ### Exceptions {#HETERO_TYPE_CONTAINER_exceptions}
160 There are three situations in which this container should throw an exception:
161 1. When the user tries to add data to the container but the type of the field is different from the
162 type of the data provided.
163 2. When the user tries to retrieve data but the type of the field is different from the type of the
164 data requested.
165 3. When the user tries to request a field that is not currently stored in the container.
166 
167 During these scenarios the container will throw an exception:
168 - in cases (1) and (2), the container will throw a ClassCastException along with a message informing
169 the user what has gone wrong.
170 - in case (3) the container will throw a NullPointerException along with a message informaing the
171 user what has gone wrong.
