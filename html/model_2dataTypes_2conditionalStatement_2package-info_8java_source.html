1 /**
2  * This Package contains the Classes necessary to replace a standard procedural conditional statement
3  * (if-then-else) with a general purpose Object Oriented version that uses double dispatch as its method
4  * of behaviour selection.
5  *
6  * ##Introduction
7  * First off this mini framework is not designed to completely replace the traditional conditional
8  * statement, while possible to do so this may adversely effect the performance of the application.
9  * It is designed for code, which for some reason reason or other, requires relatively complex
10  * condition evaluation (traditionally evaluated using if-then, switch or nested if statements.....the horror)
11  * and encapsulate and compartmentalise the various parts of these to aid in maintainability and reduce
12  * cyclomatic complexity; as well as enabling the user of this framework to have their classes BE
13  * conditional as opposed to HAVING conditionality.
14  *
15  * The use cases envisioned for this framework (although there may be other applications) are:
16  * - taking conditional statements with complex conditions and encapsulating the Condition and Result
17  * seperately.
18  * - replacing nested If statements by a 'chain of command' style pattern
19  * - Allowing users to created objects which are intrinsically conditional and to use them to simplify
20  * the design of their code.
21  *
22  * ##Getting Started, Complex Conditions (Basic Usage)
23  * Pretty much all of the use cases require similar setups so here is the basic skeleton of a test
24  * implementation:
25  *
26  * \code
27  * public class TestCondition extends Condition
28  * {
29  * private final int valueA_M;
30  * private final int valueB_M;
31  * private final int valueC_M;
32  *
33  * //all values needed for condition evaluation should be passed to constructor
34  * public TestCondition(int a, int b, int c)
35  * {
36  * this.valueA_M = a;
37  * this.valueB_M = b;
38  * this.valueC_M = c;
39  * }
40  *
41  * //defines the expression to evaluate
42  * protected boolean expression()
43  * {
44  * return this.valueA_M > this.valueB_M
45  * &&
46  * this.valueB_M > this.valueC_M;
47  * }
48  * }
49  * \endcode
50  *
51  * As you can see, the first step is to subclass the Condition class, the important method in the
52  * implementation of this class is the expression() method which returns a boolean value defined
53  * by your expression, in this case: `a > b AND b > c` . All the values needed by this condition
54  * should be passed into the constructor and stored so that this expression has data to operate on.
55  *
56  * \code
57  * public class TestResult extends Result<String>
58  * {
59  * public TestResult()
60  * {super();}
61  *
62  * //behaviour to be invoked if Condition evaluates to true goes here
63  * public String result(True yes);
64  * {
65  * return "success!";
66  * }
67  *
68  * //behaviour to be invoked if Condition evaluates to false goes here
69  * public String result(False no);
70  * {
71  * return "Failure";
72  * }
73  * }
74  * \endcode
75  *
76  * The next step is to subclass the Result class in order to define what behaviour you wish to occur
77  * given a true or false result. This is accomplished by result(True yes) and result(False no) methods
78  * which contain the respective behaviours. The Result class must be parameterised as this is how you
79  * define the return types of the result methods. i.e. if you wanted to return an integer you would
80  * parameterise it as `public class TestResult Result<Integer>` and the result methods would then
81  * also return Integer.
82  *
83  * \code
84  * public class TestConditional extends Conditional<String>
85  * {
86  * private final int valueA_M;
87  * private final int valueB_M;
88  * private final int valueC_M;
89  *
90  * //default ctor
91  * public TestConditional(int a, int b, int c)
92  * {
93  * this.valueA_M = a;
94  * this.valueB_M = b;
95  * this.valueC_M = c;
96  * }
97  *
98  * //default condition
99  * public Condition startCondition()
100  * {
101  * return new TestCondition(this.valueA_M, this.valueB_M, this.valueC_M);
102  * }
103  *
104  * //result for default condition
105  * public Result<String> startResult()
106  * {
107  * return new TestResult();
108  * }
109  * }
110  * \endcode
111  *
112  * The last stage of the setup is to define the initial conditions for the conditional. As with the
113  * similarly to the previous steps, this involves sub-classing, in this case the Conditional class.
114  * Overriding the startCondition() and startResult() methods to provide the default condition and
115  * behaviour, as well as making sure that the values needed for the Condition object (and the Result
116  * object if necessary) are passed into the constructor of the Conditional.
117  *
118  * The Conditional derived object can then be used as follows:
119  * \code
120  * final int a = 3; final int b = 2; final int c = 1;
121  *
122  * TestConditional test = new TestConditional(a,b,c)
123  *
124  * String result = test.evaluate();
125  *
126  * System.out.println()result; // prints "success"
127  * \endcode
128  *
129  * As you can see (above) it is relatively straightforward to call the default behaviour. Below is
130  * an example of passing in some non default Condition and Result objects.
131  *
132  * \code
133  * final int a = 3; final int b = 2; final int c = 1;
134  *
135  * TestCondition testCondition = new TestCondition(a,b,c);
136  * TestResult testResult = new TestResult();
137  * TestConditional test = new TestConditional();
138  *
139  * String result = test.evaluate(testCondition, testResult);
140  *
141  * System.out.println(result); // prints "Success"
142  * \endcode
143  *
144  * This ignores the default behaviour and uses the Condition and Result objects passed in directly (
145  * in this case they happen to be the same as the default behaviour, but they could be different).
146  *
147  * ##Replacing Nested If's
148  * Really speaking long sequences of if, else-if statements are a form of nested if statement, you can
149  * see from the code below that the two are equivalent:
150  *
151  * <table style="border 0px solid white"><tr><td>
152  * \code
153  * //standard If/Else-If
154  *
155  * if(condition1)
156  * {
157  * //do stuff
158  * }
159  * else if(condition 2)
160  * {
161  * //do other stuff
162  * }
163  * else if(condition 3)
164  * {
165  * //do different stuff
166  * }
167  * else
168  * {
169  * //do other different stuff
170  * }
171  * \endcode </td><td>
172  * \code
173  * //Unrolled If/Else-If
174  *
175  * if(condition1)
176  * {
177  * //do stuff
178  * }
179  * else
180  * {
181  * if(condition 2)
182  * {
183  * //do other stuff
184  * }
185  * else
186  * {
187  * if(condition 3)
188  * {
189  * //do different stuff
190  * }
191  * else
192  * {
193  * //do other different stuff
194  * }
195  * }
196  * }
197  * \endcode </td></tr></table>
198  *
199  * as a result the method for creating longer series of switch style if-else statements is the same
200  * as that for creating nested if statements. The process for this is more or less exactly the same as
201  * the basic usage with a minor modification to the Result and Conditional objects.
202  *
203  * The first step is to set up all the conditions that you will need (as described in the previous section)
204  * and then you need to set up the Result Object:
205  *
206  * \code
207  * public class NestedResult extends Result<String>
208  * {
209  * //holds the Conditional object that will handle the next set of Condition , Result object
210  * private final nextConditional<String> next_M;
211  * private final int valueA_M;
212  * private final int valueB_M;
213  * private final int valueC_M;
214  *
215  * public TestResult(Conditional<String> next, int a, int b, int c)
216  * {
217  * super();
218  * this.next_M = next;
219  * this.valueA_M = a;
220  * this.valueB_M = b;
221  * this.valueC_M = c;
222  * }
223  *
224  * //behaviour to be invoked if Condition evaluates to true goes here
225  * public String result(True yes);
226  * {
227  * return "success!";
228  * }
229  *
230  * //behaviour to be invoked if Condition evaluates to false goes here
231  * public String result(False no);
232  * {
233  * Condition nextCondition = new anotherCondition(this.valueA_M, this.valueB_M, this.valueC_M)
234  * Result<String> nextResult = new anotherResult();
235  *
236  * return this.nextHandler_M.evaluate(nextCondition, nextResult);
237  * }
238  * }
239  * \endcode
240  *
241  * Most of this should be self explanatory and very similar to what we've seen before with the Conditional
242  * Condition derived classes we've created in the 'basic usage' section, and we need it now in this
243  * result, so that it can be passed to the next Condition.
244  *
245  * You then need to set up the conditional, this is also slightly different from before but not by
246  * much:
247  * \code
248  * public class TestConditional extends Conditional<String>
249  * {
250  * private final int valueA_M;
251  * private final int valueB_M;
252  * private final int valueC_M;
253  *
254  * //default ctor
255  * public TestConditional(int a, int b, int c)
256  * {
257  * this.valueA_M = a;
258  * this.valueB_M = b;
259  * this.valueC_M = c;
260  * }
261  *
262  * //default condition
263  * public Condition startCondition()
264  * {
265  * return new TestCondition(this.valueA_M, this.valueB_M, this.valueC_M);
266  * }
267  *
268  * //result for default condition
269  * public Result<String> startResult()
270  * {
271  * return new TestResult(this, this.valueA_M, this.valueB_M, this.valueC_M);
272  * }
273  * }
274  * \endcode
275  *
276  * The only differences here is that TestConditional passes itself to the TestResult object allowing
277  * it to be used by Result objects further down the chain to evaluate other Condition and Result objects.
278  *
279  * This Conditional class is then invoked using its default behaviour and the data makes its way down
280  * the chain until it reaches a terminating outcome at which point the result makes its way back up the
281  * chain to the caller.
282  *
283  * ##Intrinsically Conditional Design
284  * This framework doesn't need to be used in isolation to replace traditional conditional statements
285  * (although it will work fine doing it this way), it can be made an intrinsic part of the class;
286  * take, for example, a problem that you chose to solve using the strategy pattern, you could make the
287  * context class implement the Conditional abstract base class allowing it to decide the particular
288  * strategy implementation at runtime, keeping the coupling between the decision logic and the strategy
289  * implementation loose; as well as keeping the cohesion high.
290  *
